**Q2: Is there more explanation about the understanding of coordinates?**

A2: The ultraArm P1 is a 4-axis robotic arm. The API for controlling coordinate movement is `set_coords([x, y, z, rz], speed)`.

**x, y, z coordinates:** Control the position of the end effector in space. Changing these values will move the end effector to different spatial positions.

**R attitude angle:** Controls the rotation of the end effector around the Z axis. The P1 has only one attitude angle rz, with a range of -180° ~ 180°. The system uses R to refer to rz.

If you want to see the changes more intuitively, we recommend using myBlockly's quick move tool, adjusting one parameter at a time and observing the changes of the end effector in space.

**Q3: Is there more explanation about the Offset of the DH parameter? Is the Offset rotated around z?**

A3: The DH parameter describes the geometric and kinematic relationship between adjacent links in the robot arm. In the DH parameter table, the Offset parameter indicates the effect of the previous link rotating around its z-axis on the position of the next link, i.e., the displacement when connecting two links. Therefore, Offset is not a rotation around z, but represents the displacement when connecting two links.

**Q4: Does the ultraArm P1 have collision detection?**

A4: Yes. The ultraArm P1 has a built-in collision detection function that monitors the robot's status in real time during motion, with millisecond-level collision response. Collision detection triggers an automatic stop and is integrated into the API for setting joint angles and coordinates.

**Q5: Is the joint torque information provided?**

A5: We disclose the overall parameters of the robotic arm, such as repeatability, power supply voltage, and payload. Detailed information such as internal torque, voltage, and current of the motor actuator unit is not provided.

**Q6: What is the voltage range of the ultraArm P1 power supply?**

A6: DC 12V, 8A.

**Q7: How do you understand the relationship between two coordinate systems in the figure below?**

A7: If you want to view the transformation relationship between a coordinate system named "turtle1" and one named "turtle2", you can use the `rosrun tf tf_echo` command. In layman's terms, when you run this command, it tells you the position and orientation information of one object ("turtle1") relative to another object ("turtle2"), just like knowing the position of one city relative to another on a map.

**Q8: The ROS2 environment has been accidentally changed. Can I delete ROS and reinstall it myself?**

A8: Regarding ROS reinstallation, we do not recommend users to reinstall it themselves, as the construction of the ROS environment is relatively complex and prone to errors. If you need to reset the ROS environment, we recommend re-flashing the system image. For specific methods, please refer to [Development Based on ROS2](../../C-FunctionsAndApplications/6.developmentGuide/6.3-ROS2/README.md).

**Q9: What is the difference between API and serial port commands for directly controlling joints?**

A9: API provides a simplified and abstract interface to make development more efficient and easy, suitable for rapid development and integration. Serial port commands provide direct, low-level control, suitable for scenarios that require fine-tuning or custom functionality, but are typically more complex to develop and debug. Overall, using serial port commands to directly control the robotic arm is more flexible but more complex, requiring a deep understanding of the communication protocol; while using API control is simpler and more convenient.

**Q11: What is the difference between MDI and JOG?**

A11: MDI (Manual Data Input) is the set-value direct-given operation mode, where the upper controller directly sets the target position, speed, and acceleration/deceleration, and the axis automatically moves to the target position. MDI is the most commonly used positioning function in practical applications. JOG moves continuously in a certain direction.

**Q12: End zero position abnormality**

A12: After using the gripper to grip objects for a long time, gripper and end zero position abnormality may occur. The gripper needs to be stationary and then re-calibrated.

**Q13: What is forward kinematics and inverse kinematics?**

A13: Forward kinematics refers to solving the position and posture of the robot's end effector in Cartesian space given the angles of each joint. It is implemented in the `get_coords_info()` API. Inverse kinematics is the opposite — solving the angles of each joint given the position and posture of the end effector in Cartesian space. It is implemented in the `set_coords()` API.
