### 命名规则

| 层级 | 规则 | 示例 |
|------|------|------|
| 一级目录 | `大写字母-英文名`（区分板块层级） | `A-ProductInformation`、`B-BasicSettings` |
| 二级目录 | CN：`数字-英文名` / <br />EN：`数字.英文名` | `1-ProductIntroduction`、`3.UserNotice` |
| 叶子文件 | `编号-英文名.md`（多级编号用 `.` 分隔） | `3.1-SafetyInstruction.md`、`5.2.1-home.md` |
| 章节入口 | `README.md` | 每二级目录都有一个 README.md，用于描述这一整章的内容。 |
| 多子页章节 | 独立目录，内含 README + 子页 | `5.2-minirobot/`、`6.1-python/` |

> **说明**：P1 使用 `A-`、`B-` 等字母前缀，目的是将一级与二级目录区分开来。450 使用数字前缀（`1-`、`2-`），功能等价。

## 二、各章节写作目的与内容说明

> 参照 450 机械臂 GitBook 各章节内容，总结每章的写作目的和要点。P1 编写时遵循相同目的，内容替换为 P1 产品实际信息。

### 产品信息

#### 1 产品介绍
**目的**：让潜在用户对产品建立全面的高层次认知，回答"这是什么产品、能做什么、适合谁用"。

**内容要点**：
- 产品概述：用一段话概括产品定位和核心价值
- 设计理念：说明产品设计的初衷和要解决的问题
- 设计目标：以表格列出目标、描述和应用场景（如教育、科研、商业）
- 产品特点：逐条列出核心卖点（如超长待机、安全关节、开源生态等），每条配简要说明
- 产品应用：列出用户群体（教育/科研/商业）及对应的应用场景，用表格区分"开箱即用"和"扩展应用"
- 支持的扩展开发：用表格列出支持的编程语言（Python/ROS/C++）、硬件接口、软件库、系统兼容性
- 购买地址：提供淘宝、Shopify、速卖通等链接

#### 2 产品特性（参数）
**目的**：为工程师和技术用户提供精确的规格数据，用于集成评估和方案设计。

**内容要点**：
- 机器人规格参数：型号、自由度、负载、工作半径、精度、重量、电源、通信协议等（表格）
- 结构尺寸参数：产品尺寸图、关节运动范围表
- 硬件电气特性：各接口（底座接口、末端接口）的实物图 + 功能说明表
- 笛卡尔坐标系：关节坐标系、用户坐标系、工具坐标系的定义和图示，DH 参数表

---

### 基础设施

#### 3 用户须知
**目的**：**必读章节**，确保用户安全正确地使用产品，明确责任划分。

**内容要点**：
- **3.1 安全须知**：危险识别（分等级表格：危险/警告/触电隐患）、安全注意事项（列表）、禁止用途
- **3.2 运输和储存**：包装要求、运输注意事项、长期储存条件、相关责任说明
- **3.3 维护和保养**：日常维护方法、不同环境下的保养建议、延长使用寿命的技巧
- **3.4 常见问题解决**：按类别（软件/硬件/其他）列出常见问题及解决方案，帮助用户快速排查

#### 4 首次安装
**目的**：引导首次使用的用户正确完成开箱到开机的全流程。

**内容要点**：
- **4.1 产品标准清单**：装箱物品列表（表格 + 实物图），核对指引
- **4.2 产品开箱指南**：开箱步骤和注意事项
- **4.3 开机检测指南**：首次上电的检测步骤，确认产品正常工作

---

### 功能与应用

#### 5 基础功能
**目的**：教会用户如何使用机械臂的基本软件工具，是操作入门核心章节。

**内容要点**：
- **5.1 系统使用说明**：系统整体概述（MiniRobot + myStudio Pro + 网络架构 + 通信接口的关系）
- **5.2 MiniRobot 介绍**：本地控制面板的完整操作指南，包含首次使用、拖动示教、BlocklyRunner、快速移动、零位校准、版本显示、无线连接、Q&A
- **5.3 myStudio Pro 介绍**：上位机编程控制软件的完整使用说明，包含首次使用、myBlockly图形化编程、快速移动、资源链接、设置、modbus、常见问题
- **5.4 零位校准**：使用 API 进行关节零位校准的方法

#### 6 软件开发指南
**目的**：为开发者提供各编程语言环境下控制机械臂的方法，是技术开发的核心参考。

**内容要点**：
- **6.1 Python**：环境搭建 → API说明 → 关节控制 → 坐标控制 → IO控制 → 夹爪控制 → 异常处理
- **6.2 ROS1**：环境搭建 → ROS基础 → RVIZ使用 → 基本功能案例 → 常见问题
- **6.3 ROS2**：环境搭建 → ROS2基础 → RVIZ2使用 → 基本功能案例 → 常见问题
- **6.4 通信协议包**：串口通信协议的格式说明和使用方法

#### 7 成功案例
**目的**：展示产品在真实场景中的应用成果，激发潜在用户的兴趣和灵感。

**内容要点**：每个案例包含场景描述、实施方案、成果展示（图文结合）。

#### 8 配套资源
**目的**：集中提供所有可下载的参考材料，方便用户一站式获取。

**内容要点**：
- 8.1 产品资料（规格书、说明书）
- 8.2 产品图纸（2D/3D 模型文件）
- 8.3 系统资料（镜像、源码）
- 8.4 宣传资料（宣传册、视频、案例）

---

### 支持与服务

#### 9 关于我们
**目的**：建立品牌信任，提供售后联系渠道。

**内容要点**：公司介绍、发展历程、相关链接（官网/视频/社交）、联系方式（邮件/微信）、购买链接。

#### 10 周边配件
**目的**：帮助用户了解和选择配件，扩展机械臂功能。

**内容要点**：配件概述 → 按类型分类（夹爪/吸泵/夹持器/其他） → 每个配件独立页面（介绍 + 参数 + 使用方法）。

#### 11 更新日志
**目的**：记录文档的版本变更历史，让用户了解文档更新内容。

**内容要点**：日期 + 更新类型（新增/优化/修改）+ 具体更新条目。

### 致谢

#### 12 致谢
**目的**：感谢参与产品开发的人员和支持者，传递品牌温度。

**内容要点**：感谢研发团队、测试团队、客户支持、合作伙伴、投资者，以及感谢用户。

## 三、分支管理与发布流程

> 依据 README.md 中的开发规范。

1. 基于 `develop` 分支创建/拉取新分支进行编写
2. 编写完成后先合并至 `develop`
3. 由产品助理检查后统一合并至主支（`main`）
4. 禁止直接向 `main` 提交

## 四、资源引用规范

### 4.0 资源存放架构

**图片资源统一存放在仓库根目录 `resources/` 下，供中文版（`ultraArmP1_cn/`）和英文版（`ultraArmP1_en/`）共享**。

```
ultraArmP1_docs/
├── resources/                          # ← 共享资源目录（唯一）
│   ├── A-ProductInformation/
│   ├── B-BasicSettings/
│   ├── C-FunctionsAndApplications/
│   ├── D-SupportAndService/
│   └── E-Acknowledgments/
├── ultraArmP1_cn/                      # 中文版（不含 resources）
└── ultraArmP1_en/                      # 英文版（不含 resources）
```

> **设计理由**：中英文文档引用的图片（产品照片、UI 截图、图纸等）完全相同，不应重复存放两份。将 `resources/` 提升到仓库根目录，中英文版本通过相对路径共同引用。

### 4.1 图片尺寸

| 类型 | 宽度 | 适用场景 |
|------|------|------|
| 大图 | `"1200"` | 产品展示、全宽截图 |
| 中图 | `"550"` | 功能说明、界面截图 |
| 小图 | `"300"` | 单个小组件、小屏幕截图 |

> 若需采用其他尺寸，截图向产品确认效果后方可使用。

### 4.2 图片命名与存放

- 图片名称统一使用**英文**
- 图片存放路径：仓库根目录 `resources/` 下，镜像 docs 目录结构
- **图片资源目录最深为三级**（第三级为 `X.X` 层级），更深层级的章节图片直接放在所属三级目录中

**示例**：`5.2.1` 章节的图片放在：
```
resources/C-FunctionsAndApplications/5-BasicApplication/5.2-minirobot/
```
不再往下新建 `5.2.1/` 子目录。

### 4.3 图片引用格式

引用路径使用**相对路径**，从 Markdown 文件所在位置向上回溯到仓库根目录的 `resources/`。`../` 数量取决于文件在 `ultraArmP1_cn/`（或 `ultraArmP1_en/`）内的深度：

| 文件深度 | `../` 数量 | 引用示例 |
|---------|-----------|---------|
| 二级目录下（如 `1-ProductIntroduction/`） | 3 | `../../../resources/A-ProductInformation/1-ProductIntroduction/xxx.png` |
| 三级目录下（如 `5.2-minirobot/`） | 4 | `../../../../resources/C-FunctionsAndApplications/5-BasicApplication/5.2-minirobot/xxx.png` |
| 四级目录下（如 `6.1-python/`） | 4+ | `../../../../resources/C-FunctionsAndApplications/6-SoftwareDevelopment/6.1-python/xxx.jpg` |

**格式**：
```markdown
<img src = ../../../resources/对应目录/图片名.png
width = "1200" align = "center">
```

| 规则 | 说明 |
|------|------|
| 路径 | 使用相对路径，`../` 数量 = 文件在 CN/EN 内的目录层级数 + 1 |
| 宽度 | 大图 `"1200"`、中图 `"550"`、小图 `"300"` |
| 对齐 | 统一 `align = "center"` |
| 格式 | PNG 优先 |

## 五、SUMMARY.md 编写规范

### 4.1 格式

```markdown
# Summary

* [Introduction](README.md)

## 一级分类名
* [X 章节名](路径/README.md)
    * [X.X 小节名](路径/文件名.md)
        * [X.X.X 子页](路径/文件名.md)
```

- 一级分类使用 `##` 标题分组
- 每个缩进级别用 4 个空格
- 显示名称用中文（中文版）或英文（英文版）
- 链接路径为相对于 SUMMARY.md 的路径

### 4.2 目录深度

显示到第三级（如 `5.2.1`），更深内容通过页面内部导航处理。

## 六、编写流程

1. 基于 `develop` 分支创建新分支
2. 在 `ultraArmP1_cn` 中编写中文版
3. 中文版定稿后翻译为英文版放入 `ultraArmP1_en`
4. 更新两侧 `SUMMARY.md` 保持同步
5. 本地 `gitbook build` 验证渲染效果
6. 合并至 `develop`，由产品助理检查后合并至 main
